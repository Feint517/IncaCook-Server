# Signup flow

How users become buyers, sellers, or drivers on IncaCook — and how the
backend should model the data that gets collected along the way.

This doc has two halves. The first describes the wizard as the user
experiences it, screen by screen, and the two backend calls already
wired ([`/v1/auth/signup`](./flutter-integration.md#4-auth-flow) and
`POST /v1/users`). The second proposes how to handle the rest of the
data — the role-specific stuff that today still lives only in client
memory — via resource-per-concept endpoints plus a single
`GET /v1/users/me/onboarding` endpoint that acts as the resume cursor
and the source of truth for "is this user done?".

For the underlying HTTP conventions (envelopes, error shape,
idempotency, refresh) see [`flutter-integration.md`](./flutter-integration.md).

---

## 1. What the user sees

The wizard is a single `PageView` driven by
[`SignupFlowController`](../lib/features/authentication/controllers/signup_flow_controller.dart).
The step list is computed dynamically from the chosen role / category /
vehicle, so users only see screens that apply to them.

### 1.1 Common preamble — all roles

Every signup starts with the same five screens:

| # | Step | What's collected |
|---|---|---|
| 1 | Basic info | first name, last name, email, phone, password (with strength meter) |
| 2 | Phone verification | 6-digit OTP |
| 3 | Biometric setup | opt-in Face/Touch ID (device-local, never synced) |
| 4 | Legal acceptance | CGU + CGV checkboxes |
| 5 | Role selection | buyer / seller / driver |

After role selection, the wizard branches.

### 1.2 Buyer branch

| # | Step | What's collected |
|---|---|---|
| 6 | Delivery address | line1, city, postcode, lat/lng (Mapbox-geocoded) |
| 7 | Dietary & allergies | dietaryTags[], allergens[] — **skippable** |
| 8 | Done | confirmation, no inputs |

### 1.3 Seller branch

| # | Step | What's collected |
|---|---|---|
| 6 | Profile | photo URL, display name, bio |
| 7 | DOB & pickup address | date of birth (must be ≥18), pickup address |
| 8 | Business info | business name, SIRET (Luhn-checked), facade photo, weekly opening hours — **shown only for non-`fait_maison` categories** |
| 9 | Cuisine | cuisineTypes[], dishTypes[] |
| 10 | KYC — ID | document type, front photo, back photo (when the type requires verso) |
| 11 | KYC — selfie | selfie holding the ID |
| 12 | Charter | hygiene commitment + fait-maison commitment |

### 1.4 Driver branch

| # | Step | What's collected |
|---|---|---|
| 6 | DOB & address | date of birth (must be ≥18), home address |
| 7 | Vehicle | vehicle type (bike / scooter / car / …) |
| 8 | KYC — ID | document type, front, back |
| 9 | KYC — selfie | selfie with ID |
| 10 | Vehicle documents | driving license, carte grise — **shown only when vehicleType.requiresMotorizedDocs** |
| 11 | Operating zones | zone identifiers |
| 12 | Charter | punctuality commitment + care commitment |

### 1.5 Dynamic skipping

Three branches in the step list can shorten the wizard:

- `seller.business_info` is skipped for the fait-maison category (no SIRET needed).
- `driver.documents` is skipped for non-motorized vehicles.
- `buyer.dietary` is user-skippable from the bottom bar (the "Passer" button).

The skip logic lives in the controller; the backend should treat any
skipped step's absence as `skipped` (not `incomplete`) when deriving
completeness — see §4.

## 2. Backend gates today

Of the screens above, only two trigger network calls. Everything else
collects into reactive client-side state and is sent later, in batches
that match real backend resources (see §3).

### 2.1 Gate 1 — `POST /v1/auth/signup`

Fires when the user passes the Basic Info validator and taps Continue.

```http
POST /v1/auth/signup
Content-Type: application/json

{ "email": "...", "password": "..." }
```

Returns a `Session` (access + refresh tokens, expiresAt, auth user).
The repository persists tokens to `flutter_secure_storage` before
returning, so every subsequent request in the wizard carries
`Authorization: Bearer <accessToken>`.

The controller flips an `isSignedUp` flag — navigating back to Basic
Info and forward again does **not** double-call this endpoint.

### 2.2 Gate 2 — `POST /v1/users`

Fires after the user picks a role.

```http
POST /v1/users
Authorization: Bearer <accessToken>
Idempotency-Key: <ULID>
Content-Type: application/json

{
  "firstName": "...",
  "lastName": "...",
  "role": "buyer" | "seller" | "driver",
  "cguAccepted": true,
  "cgvAccepted": true
}
```

Creates the IncaCook profile row. From this point on the user is a
"real" buyer/seller/driver — RLS policies that key off `users.role`
start applying.

The idempotency key is auto-generated by the
[`ApiClient`](../lib/core/network/api_client.dart) per Continue press,
so retrying after a network blip will never create a duplicate row.

### 2.3 What about everything else?

After Gate 2, the user proceeds through six to seven more screens
collecting role-specific data — addresses, KYC, business info, cuisines,
vehicles, charters, preferences. None of it currently leaves the device.

The rest of this document is the proposal for how the backend should
receive that data.

## 3. Resource-per-concept endpoints

We considered three shapes for sending the role-specific data and
picked the third:

| Shape | Pros | Cons |
|---|---|---|
| One fat `POST /v1/signup/complete` at the end | Simple client | A failed KYC upload nukes the whole submission. No partial resume. |
| `POST /v1/signup/steps/:step` for each step | Maps 1:1 to the wizard | Couples the API to wizard order. Future edit screens need a different surface. |
| **Resource-per-concept endpoints** | Same endpoint serves the wizard and post-signup editing. Granular retries. RLS targets stay clean. | More endpoints to define and secure. |

Each role-specific concept maps to one table and one endpoint pair
(read + upsert).

### 3.1 Tables

| Table | Owned by | Holds |
|---|---|---|
| `users` | Gate 2 | id, email, role, firstName, lastName |
| `seller_profiles` | seller | displayName, bio, photoUrl, dob, sellerCategory |
| `seller_businesses` | seller (optional) | businessName, siret, facadeUrl, openingHours |
| `seller_cuisines` | seller (join) | userId × cuisineType / dishType |
| `buyer_preferences` | buyer | dietaryTags[], allergens[] |
| `driver_profiles` | driver | dob, vehicleType |
| `driver_zones` | driver (join) | userId × zoneId |
| `addresses` | shared | userId, kind (`buyer_delivery` / `seller_pickup` / `driver_home`), line1, lat, lng |
| `kyc_documents` | shared | userId, type (`id_front` / `id_back` / `selfie` / `driving_license` / `carte_grise`), fileUrl, reviewState |
| `user_charters` | shared | userId, charter (`hygiene` / `fait_maison` / `punctuality` / `care`), version, acceptedAt |

### 3.2 Endpoints

```
PUT   /v1/sellers/me/profile          { displayName, bio, photoUrl, dob, category }
PUT   /v1/sellers/me/business         { businessName, siret, facadeUrl, openingHours }
PUT   /v1/sellers/me/cuisines         { cuisines:[...], dishTypes:[...] }
PUT   /v1/buyers/me/preferences       { dietaryTags:[...], allergens:[...] }
PUT   /v1/drivers/me/vehicle          { vehicleType }
PUT   /v1/drivers/me/zones            { zones:[...] }
PUT   /v1/users/me/addresses/:kind    { line1, line2, city, postcode, country, lat, lng }
POST  /v1/kyc/documents               (multipart) { type, file }       → { id, reviewState }
POST  /v1/users/me/charters           { charter, version }             → { acceptedAt }
GET   /v1/users/me/onboarding                                          → see §4
```

**PUT** for most because each is keyed by `(userId, kind)` — repeat
calls upsert, so retries are idempotent without an explicit
`Idempotency-Key`. **POST** only where each call genuinely appends a
row (KYC documents — one row per uploaded photo; charter acceptance —
one row per acknowledgement, so we can track history).

KYC stays orthogonal to roles. The same `POST /v1/kyc/documents`
endpoint accepts seller selfies and driver carte-grises — the `type`
enum decides which buckets are required for which role. The completeness
endpoint (§4) is what tells the wizard which buckets it still needs.

Per the IncaCook RLS convention, KYC pending state is a
listing-visibility gate, not a UI block: sellers and drivers proceed
through the wizard and into their role home immediately after the
charter step. Whether their listings appear publicly / they get
matched to orders is decided server-side by RLS policies reading
`kyc_documents.reviewState`, not by a "waiting for approval" screen.

## 4. The completeness endpoint

Granular endpoints solve "how do I save each piece," but raise a new
question: **who decides the user is done?**

Today, Flutter knows — the wizard's `canGoNext()` validator at each
step contains the rules. But that knowledge will soon need to live in
more places:

- The eventual web signup flow.
- The admin "approve this seller" tooling (which needs to know what's
  still missing, not just whether KYC was uploaded).
- The post-signup banner system — `PayoutSetupBanner` already exists
  for Stripe Connect onboarding; future banners for re-do-your-KYC,
  add-a-second-vehicle, etc. will all key off completeness state.
- The RLS policies that gate listings on KYC review.

Three sources of truth is one too many. We expose one:

```http
GET /v1/users/me/onboarding
```

### 4.1 Response shape

```jsonc
{
  "role": "seller",
  "next": "kyc_selfie",
  "steps": {
    "profile":     "complete",
    "business":    "skipped",
    "cuisines":    "complete",
    "addresses":   "complete",
    "kyc_id":      "complete",
    "kyc_selfie":  "incomplete",
    "charter":     "incomplete"
  },
  "kycReviewState": "pending",
  "canList":    false,
  "canDeliver": false
}
```

### 4.2 Status values

| Status | Meaning |
|---|---|
| `complete` | Row exists with all required fields. |
| `incomplete` | Row missing or partially filled. |
| `skipped` | Explicitly not required for this user (e.g. `business` for fait-maison sellers, `documents` for non-motorized drivers). |
| `pending_review` | Submitted but awaiting admin action. Currently only relevant for KYC documents. |

### 4.3 Derivation rules

The server computes each `steps.*` from underlying table state. The
rules are role-specific but each is a single deterministic check:

| Step key | Status logic |
|---|---|
| `profile` | `seller_profiles` row exists with photoUrl + displayName + dob |
| `business` | `seller_businesses` row exists, OR sellerCategory is fait-maison → `skipped` |
| `cuisines` | `seller_cuisines` has ≥1 cuisine row |
| `addresses` | `addresses` row exists for the role's required `kind` |
| `kyc_id` | `kyc_documents` rows exist for `id_front` (+ `id_back` when the doc type requires verso) |
| `kyc_selfie` | `kyc_documents` row exists for `selfie` |
| `charter` | All role-required charters acknowledged in `user_charters` |
| `vehicle` | `driver_profiles.vehicleType` is set |
| `zones` | `driver_zones` has ≥1 row |
| `documents` | If `vehicleType.requiresMotorizedDocs` then license + carte_grise rows exist, else `skipped` |

The `next` field is the first step in role-canonical order whose
status is `incomplete`. When every step is `complete` or `skipped`,
`next` is `null`.

The two derived booleans bundle the rules downstream code actually
needs:

- **`canList`** (seller) — every step `complete` or `skipped` AND
  `kycReviewState != rejected`. Listings RLS reads this.
- **`canDeliver`** (driver) — analogous. Driver-assignment RLS reads
  this.

Buyer never has either flag — buyers can browse and order immediately
after Gate 2; addresses and preferences are convenience, not gates.

### 4.4 Client usage

The wizard calls `GET /users/me/onboarding` in two situations:

**Cold start with a session but no profile / incomplete profile.**
The Flutter app reads tokens from secure storage on launch; if they
exist, it fetches `/users/me/onboarding`. A non-null `next` means the
user abandoned mid-signup — the app jumps the `PageView` to that step.

```dart
final state = await usersRepository.fetchOnboarding();
if (state.next != null) {
  controller.currentPage.value = controller.steps.indexWhere(
    (s) => s.key == state.next,
  );
}
```

**After each role-specific PUT/POST**, to confirm the status flipped
and to learn the new `next` (which may have skipped ahead past an
optional step).

This means the wizard's step list **does not** need to encode "which
steps are required for which role" — that knowledge lives server-side
in the derivation rules. The Flutter step list just defines the UI
order; completeness defines the truth.

### 4.5 Resume across devices

Because the completeness endpoint is the single source of truth, a
user who starts signup on iOS, gets to KYC, then signs in on the web
will land on the same screen they left iOS at. No client-side
synchronization is needed — both surfaces consume the same shape.

## 5. Lifecycle summary

The end-to-end flow, expressed as the sequence of state transitions
the backend observes:

```
1.  POST  /v1/auth/signup                    → row in auth.users
                                               (session returned)
2.  POST  /v1/users                          → row in users
                                               (Gate 2, role committed)
3.  PUT   /v1/sellers/me/profile             ┐
    PUT   /v1/sellers/me/business            │  Role-specific tables
    PUT   /v1/sellers/me/cuisines            │  populate in whatever order
    PUT   /v1/users/me/addresses/seller_pickup │  the user advances.
    POST  /v1/kyc/documents (id_front)        │  Each call upserts its
    POST  /v1/kyc/documents (id_back)         │  own row; the user can
    POST  /v1/kyc/documents (selfie)          │  abandon and resume.
    POST  /v1/users/me/charters (hygiene)    │
    POST  /v1/users/me/charters (fait_maison)┘
4.  Each PUT/POST is followed by GET /v1/users/me/onboarding,
    which returns the next incomplete step. When `next == null`,
    the wizard navigates to the role home.
5.  PayoutSetupBanner (sellers) checks for an active Stripe Connect
    Express account; if missing, prompts onboarding. This is
    independent of /onboarding completeness — it's the next
    post-signup gate, not a wizard step.
```

## 6. Open questions

A few small decisions need locking before the backend lands. None of
them block the Flutter implementation — typed repos and the
`OnboardingState` model can be written against the shape in §4.1 today.

### 6.1 Skipped steps — empty PUT or no call at all?

When the wizard skips `business` (fait-maison) or `documents` (bicycle
driver), should the client send `PUT` with an empty body to mark the
step skipped, or simply not call the endpoint and let the server
derive `skipped` from absence + role rules?

**Recommendation:** the latter. The client stays unaware of "which
steps are optional for which role" — that knowledge lives once,
server-side, in the derivation table from §4.3.

### 6.2 Charter versioning

The French market requires us to record *which version* of the
hygiene charter the seller accepted (so we can re-prompt on a version
bump and prove what was agreed to). The proposed `user_charters` row
shape (`{ charter, version, acceptedAt }`) handles this, but the
client needs to know the current version to send. Options:

- Hardcode in the Flutter app, bump on every redeploy.
- Expose `GET /v1/charters/active` returning `{ hygiene: "v1.3",
  fait_maison: "v1.0", ... }`.

The second is one extra call but lets us roll out charter updates
without an app release.

### 6.3 KYC upload mechanism

Direct multipart to `/v1/kyc/documents`, or two-step (`POST /v1/uploads`
returns a pre-signed Supabase Storage URL → client uploads directly →
`POST /v1/kyc/documents` with the resulting URL)?

**Recommendation:** two-step. Mobile bandwidth retries are friendlier
when the file upload is independent of the metadata write. Also
cleaner for resumable uploads on large selfies / facade photos.

### 6.4 OTP and phone verification

The phone-verification step (`POST /auth/otp/request`, `POST /auth/otp/verify`?)
isn't in the [`flutter-integration.md`](./flutter-integration.md)
endpoint catalog yet. The Flutter side is stubbed against
[`SignupRepository.sendOtp`](../lib/features/authentication/data/repositories/signup_repository.dart)
and accepts `123456` in dev. The backend surface needs to land before
phone verification becomes a real gate — until then it's purely
local-state, not part of the completeness model.
